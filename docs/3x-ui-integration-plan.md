# План интеграции с 3x-ui

Документ сопоставляет возможности проекта **[3x-ui](https://github.com/MHSanaei/3x-ui)** с текущим функционалом **zetep-panel (`z-panel`)** и предлагает поэтапную дорожную карту. Факты о кодовых базах проверены по открытым источникам (README, Wiki, `go.mod`) и по дереву репозитория `/var/www/home/server/z-panel`.

---

## 1. Краткий технический обзор 3x-ui


| Аспект                                                                   | Содержание                                                                                                                                                                                                                                                                                                                                                                  |
| ------------------------------------------------------------------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Назначение**                                                           | Веб-панель управления сервером **Xray-core**: настройка входящих подключений (inbounds), клиентов, шаблонов конфигурации, мониторинг и ограничения.                                                                                                                                                                                                                         |
| **Язык и модульность**                                                   | **Go** (`module github.com/mhsanaei/3x-ui/v2`, в актуальной ветке указана версия Go **1.26.x** в `go.mod`).                                                                                                                                                                                                                                                                 |

| **Данные** (именно **3x-ui**)                                            | **GORM** + драйвер **SQLite** (`gorm.io/gorm`, `gorm.io/driver/sqlite`) — всё состояние панели там: пользователи панели, inbounds, клиенты, лимиты и т.п. (**это не относится к z-panel** — см. §2.1.)                                                                                                                                                                                                        |
| **Xray**                                                                 | Прямая зависимость `**github.com/xtls/xray-core`** — панель управляет жизненным циклом и конфигурацией ядра на хосте.                                                                                                                                                                                                                                                       |
| **Доп. возможности** (по [Wiki](https://github.com/MHSanaei/3x-ui/wiki)) | Мультипротокольные inbounds (VMess, VLESS, Trojan, Shadowsocks, Tunnel, Mixed, HTTP, Wireguard и др.), XTLS/Reality, статистика трафика и лимиты, поиск по inbounds/clients, HTTPS/SSL и автоматическое обновление сертификатов, экспорт/импорт БД, обновление geo-файлов, Telegram-бот; в зависимостях также **LDAP**, **cron**, QR-коды, мониторинг системы (`gopsutil`). |
| **Лицензия**                                                             | **GPL-3.0** (указано в README).                                                                                                                                                                                                                                                                                                                                             |


Итого: 3x-ui — это **серверное приложение с веб-интерфейсом и SQLite**, тесно связанное с **Xray-core** как с управляемым процессом и конфигом.

## 2. Текущее состояние z-panel

### 2.1. Архитектурное требование: единый файл конфигурации и секреты

**Канонический источник настроек z-panel** — один файл **`/etc/z-panel/config.toml`**, задаваемый константами `ConfigDir` / `ConfigFile` в `internal/config/config.go`; отдельного «основного» пути конфигурации в репозитории нет. Если файла нет, действуют встроенные значения по умолчанию, затем при наличии файла они дополняются/перезаписываются содержимым TOML (логика в `internal/settings/settings.go`). Каталог **`/etc/z-panel/`** служит только для этого дерева конфигурации (и создаётся при `config init`), а не для произвольного набора конфиг‑файлов с равным статусом.

**Переменные окружения не заменяют этот файл:** они относятся к режиму запуска, локали и транспорту (например `Z_PANEL_LANG`, `Z_PANEL_SKIP_DAEMON`, `Z_PANEL_NO_BANNER`, параметры удалённого SSH вида `Z_PANEL_SSH_HOST` / `Z_PANEL_SSH_MUX` и т.п. — см. код и пользовательские документы по флагам SSH). Структурные поля (`state_dir`, маршрутизация, daemon, язык после загрузки и т.д.) по-прежнему считаются частью канона **`config.toml`**.

**Секреты** (пароли, долгоживущие токены и аналоги, если они появятся в конфигурации продукта): **требование** — в файле хранить только **хэши** с современными схемами вроде **bcrypt** или **Argon2** (выбор параметров и точный формат полей — задача реализации; открытый текст в конфиге не допускать).

**Отличие от x-ui / 3x-ui:** у панели на сервере учёт пользователей, inbounds и большая часть «настроек панели» живут в **SQLite** приложения **3x-ui**. **z-panel БД не использует:** любые параметры продукта, которые нужно сохранять между запусками, живут в **конфиге** (канонический **`config.toml`**; при необходимости — смежные секции/ключи там же или согласованные файлы под **`/etc/z-panel/`**, без отдельной СУБД). Дисковое состояние поднятых интерфейсов (`xray-redirect`) допускается в **JSON** внутри **`state_dir`**, задаваемого из конфигурации — это не БД домена учёта клиентов, а снимки для идемпотентного `down`/диагностики.

| Аспект                 | Содержание                                                                                                                                                                                                                                                                                                       |
| ---------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Назначение**         | **CLI для Linux** для настройки **политик маршрутизации через TUN** в стиле wg-quick: дефолтный маршрут через выбранный интерфейс, опциональный обход через **cgroup**, правила фаервола против утечек, помощники для **UFW** и адресации **systemd-networkd** для TUN. См. корневой `README.md`.                |
| **Стек**               | **Go 1.21**, единственная внешняя зависимость в `go.mod`: `**github.com/pelletier/go-toml/v2`**.                                                                                                                                                                                                                 |
| **Команды**            | Регистрируются в `main.go`: `version`, `install`, `install-shell`, `config`, `daemon`, `xray-redirect`, `ufw`, `xray-tun` (`internal/app/registry.go`).                                                                                                                                                          |
| **Конфигурация**       | Единственный канонический TOML: `/etc/z-panel/config.toml` (путь зафиксирован в `internal/config/config.go`; загрузка/дефолты — `internal/settings/settings.go`): таблица маршрутизации, каталог состояния, маркеры UFW/Xray TUN, LAN по умолчанию, адрес/peer для Xray TUN по умолчанию, путь Unix-сокета, режим daemon, язык. **Никаких GORM/SQLite и прочих СУБД:** учёт домена приложения только через конфиг (и см. строку про `state_dir`). Секреты в конфиге — только хэши (§2.1). |
| **Состояние на диске** | JSON-файлы состояния для поднятого `xray-redirect` в `state_dir` (`internal/state/state.go`).                                                                                                                                                                                                                    |
| **«API»**              | Не публичный REST для домена VPN; локальный **HTTP поверх Unix-сокета** в пакете `internal/daemon`: `GET /v1/health`, `POST /v1/run` с телом JSON `{ "args": [...] }` — выполняется тот же бинарник с переданными аргументами (`internal/daemon/server.go`).                                                     |
| **Аутентификация API** | **Отсутствует**: защита опирается на права ОС (сокет с правами **0600**, см. `Serve` в `server.go`). Команда `daemon` запрещена для удалённого выполнения через сокет (`ForbiddenRemote`).                                                                                                                       |
| **Клиентский режим**   | При `daemon=1` в конфиге CLI пересылает подкоманды на сокет (`internal/client/client.go`).                                                                                                                                                                                                                       |
| **Удалённость**        | Запуск через SSH: режимы «удалённый z-panel» и «локальные утилиты по SSH» (`main.go`, `internal/transport`). Часть команд при удалённом режиме запрещена (в т.ч. `daemon`, `xray-tun`, `xray-redirect`).                                                                                                         |
| **Связь с Xray**       | **Непрямая**: подкоманды `**xray-redirect`** и `**xray-tun**` настраивают **сеть ОС** (маршруты, nftables/ip правила, drop-in для systemd-networkd), а **не** парсят и не генерируют JSON/API конфигурацию Xray и не управляют процессом Xray. Имена отражают типичный сценарий использования вместе с Xray TUN. |
| **Лицензия**           | **GPL-2.0-or-later** (корневой README).                                                                                                                                                                                                                                                                          |


**Вывод:** z-panel и 3x-ui решают **разные задачи в одной экосистеме** (клиентская политика маршрутизации vs серверная панель Xray). Пересечение по коду минимально: оба проекта на **Go**; **z-panel** сохраняет данные приложения только в **конфигурационных файлах** (+ JSON состояния в `state_dir` при необходимости), без реляционной БД; **3x-ui** для своей панели использует **SQLite**; управления Xray-core в z-panel **нет**.

## 3. Матрица функций: 3x-ui → z-panel → подход к интеграции

Условные обозначения статуса в z-panel: **нет** | **частично** | **есть (иное назначение)**.


| Возможность 3x-ui                                      | Статус в z-panel                                    | Предлагаемый подход интеграции                                                                                                                                                                                    |
| ------------------------------------------------------ | --------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Веб-интерфейс панели                                   | нет                                                 | Не встраивать в z-panel как монолит без отдельной архитектуры: либо отдельный сервис (сам 3x-ui), либо новый веб-слой поверх существующих операций (отдельный репозиторий/бинарник).                              |
| Управление процессом и конфигом **Xray-core**          | нет                                                 | Поставить **3x-ui** (или другой контроллер Xray) на сервере; z-panel не дублировать. При необходимости единого UX — ссылки/скрипты развёртывания, не смешивание кодовых баз без аудита лицензий (GPL-2 vs GPL-3). |
| Inbounds / мультипротокол                              | нет                                                 | Только через sidecar **3x-ui** или собственный будущий модуль (высокая стоимость).                                                                                                                                |
| Клиенты, подписки, QR                                  | нет                                                 | Вне scope текущего z-panel; генерация подписок остаётся в панели типа 3x-ui.                                                                                                                                      |
| Маршрутизация **внутри Xray** (routing rules, geosite) | нет                                                 | Настраивается в конфиге Xray на стороне сервера/клиента; z-panel может документировать совместимые сценарии **без** редактирования json Xray.                                                                     |
| Статистика трафика, лимиты, срок действия              | нет                                                 | Реализовано в 3x-ui + SQLite; в z-panel не повторять без явного продукта «учёт пользователей». Для самого z-panel по-прежнему только конфиг (+ `state_dir`), **без введения БД** (§2.1).                                                                                                                                           |
| Мульти-пользователь (учётки панели)                    | нет                                                 | z-panel ориентирован на root/локальный сокет; мульти-тенантность потребовала бы модели прав и аутентификации — отдельный проектный этап.                                                                          |
| Мониторинг состояния системы                           | нет                                                 | Опционально: интеграция только как чтение метрик с хоста или делегирование внешнему стеку; не смешивать с текущим daemon без ТЗ.                                                                                  |
| Telegram-бот, LDAP                                     | нет                                                 | Зона ответственности 3x-ui или отдельных сервисов.                                                                                                                                                                |
| HTTPS / SSL для панели                                 | нет                                                 | Относится к веб-панели (3x-ui + reverse proxy).                                                                                                                                                                   |
| **Policy routing / дефолт через TUN / анти-утечка**    | **есть** (`xray-redirect`, firewall, cgroup bypass) | Сохранить как основной вклад z-panel; в документации явно описать связку «**3x-ui на сервере + z-panel на клиенте**».                                                                                             |
| **Адресация TUN (systemd-networkd)**                   | **есть** (`xray-tun`)                               | То же: клиентский хост; не конфликтует с 3x-ui.                                                                                                                                                                   |
| **UFW-хелперы**                                        | **есть** (`ufw`)                                    | Независимо от 3x-ui.                                                                                                                                                                                              |
| Локальный IPC (Unix socket)                            | **есть** (daemon)                                   | Возможная точка расширения: вызов из оркестратора **осторожно** (безопасность сокета, не экспонировать в сеть).                                                                                                   |
| Установка бинарника, `config init`                     | **есть**                                            | Остаётся каналом доставки клиентской части.                                                                                                                                                                       |

## 4. Поэтапный план внедрения

### Фаза 0 — Продуктовое решение и границы (1–2 недели)

- Зафиксировать цель: **«единый продукт»** vs **«рекомендуемый стек: 3x-ui + z-panel»** vs **«форк/встраивание»**.
- Провести **обзор лицензий**: GPL-3 (3x-ui) и GPL-2+ (z-panel) при совместном распространении/линковке — с юридической точки зрения.
- Зафиксировать модель угроз: текущий daemon **без аутентификации** не должен быть доступен непривилегированным процессам на многопользовательском хосте без дополнительных мер.

### Фаза 1 — Документация и сценарии совместной работы (низкий риск)

- Добавить в пользовательскую документацию раздел «Совместное использование с 3x-ui»: сервер (панель + Xray), клиент (Linux + z-panel для `xray-redirect` / `xray-tun`).
- Описать переменные окружения/флаги, уже используемые в z-panel (`Z_PANEL_SKIP_DAEMON`, удалённый SSH и ограничения) — явно отделив их от канонического **`/etc/z-panel/config.toml`** (§2.1: env задают режим, а не второй файл настроек).

### Фаза 2 — Операционная интеграция (sidecar)

- Скрипты или unit-файлы: порядок запуска **Xray → z-panel xray-redirect up** (или наоборот согласно стабильности интерфейса).
- Чеклист проверки утечек (маршруты, марки cgroup, UFW).

### Фаза 3 — Программная склейка (опционально, средний риск)

- Тонкая обёртка: вызов HTTP API 3x-ui (если используется его API) из **отдельной** утилиты или подкоманды — **не** смешивать с daemon z-panel без аутентификации и модели доверия; параметры узла z-panel сохранять в каноническом **`/etc/z-panel/config.toml`** без параллельного «равного по статусу» файла приложения (§2.1).
- Либо реверс-прокси и SSO на уровне инфраструктуры (вне репозитория z-panel).

### Фаза 4 — Глубокая интеграция (высокий риск, только при явном ТЗ)

- Перенос функций панели внутрь z-panel или наоборот: огромный объём, типичное дублирование связки **SQLite + веб-стека + управление Xray** в одном бинарнике, поддержка и безопасность. Для z-panel сохранять принцип **без БД**: расширения функций — через **расширение конфига**/смежные артефакты под тем же деревом **`/etc/z-panel/`**, а не через GORM/SQLite для доменных данных; **единый файл настроек** **`/etc/z-panel/config.toml`** и **секреты только хэшами** остаются обязательными (§2.1).
- Альтернатива: общий **мета-репозиторий** (compose/ansible) без слияния исходников.

### Зависимости между фазами

```text
Фаза 0 (границы, лицензии)
    ↓
Фаза 1 (документация сценариев)
    ↓
Фаза 2 (sidecar / эксплуатация)
    ↓
Фаза 3 (опционально: API-обёртки, инфраструктура)
    ↓
Фаза 4 (опционально: глубокое слияние — только при ТЗ)
```

### Риски


| Риск                                          | Митигация                                                                                         |
| --------------------------------------------- | ------------------------------------------------------------------------------------------------- |
| Путаница ролей (панель vs клиентский роутинг) | Чёткая терминология в документации и матрице выше.                                                |
| Безопасность Unix-сокета daemon               | Оставить доступ только root; не слушать TCP; при multi-user — добавить auth или отключить daemon. |
| Лицензионная несовместимость при слиянии кода | Юридическая проверка до смешивания репозиториев.                                                  |
| Дублирование функций 3x-ui в z-panel          | Избегать; переиспользовать 3x-ui как серверную панель.                                            |

## 5. Целевая архитектура после «полной интеграции» (рекомендуемая, без слияния кодовых баз)

Рекомендуемая целевая картина — **разделение ответственности** и оркестрация на уровне документации/деплоя, а не один монолит.

```text
┌─────────────────────────────────────────────────────────────────┐
│                        Сервер (VPS / дом)                        │
│  ┌──────────────┐    управляет     ┌─────────────────────────┐  │
│  │   3x-ui      │ ───────────────► │ Xray-core (процесс/конфиг)│  │
│  │ (Web + SQLite)│ ◄──────────────► │ inbounds / routing / TLS │  │
│  └──────────────┘                  └─────────────────────────┘  │
└─────────────────────────────────────────────────────────────────┘
                              │
                              │ VPN/TUN трафик
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│                   Клиент (Linux, рабочая станция)               │
│  ┌──────────────┐   поднимает политику   ┌────────────────────┐ │
│  │  Xray (TUN)  │ ◄────────────────────► │ маршруты / firewall │ │
│  └──────────────┘                        └────────────────────┘ │
│         ▲                                      ▲                 │
│         │                                      │                 │
│         └────────────── z-panel ──────────────┘                 │
│              xray-tun / xray-redirect / ufw                     │
│         state JSON (state_dir); единый /etc/z-panel/config.toml §2.1         │
│              опционально: daemon (Unix socket, POST /v1/run)    │
└─────────────────────────────────────────────────────────────────┘
```

**Границы данных:**


| Компонент | Данные                                                                                                           |
| --------- | ---------------------------------------------------------------------------------------------------------------- |
| 3x-ui     | SQLite: учётные записи панели, inbounds, клиенты, лимиты, настройки панели; конфиги Xray под управлением панели. |
| Xray      | JSON-конфигурация, runtime состояние соединений.                                                                 |
| z-panel   | Только конфиг (**без СУБД**): один канонический **`/etc/z-panel/config.toml`**; при необходимости — JSON состояния интерфейсов в **`state_dir`** (`internal/state`); команды ядра через `ip`, nftables и т.д.; любые секреты в конфиге — только в виде хэшей (**bcrypt** / **Argon2** и т.п., §2.1).                |


**Границы доверия:** панель 3x-ui — **сетевой сервис** (TLS, пароли, возможно LDAP); z-panel daemon — **локальный root IPC**, не предназначен для публикации в LAN без доработки.

## 6. Итоговая рекомендация

Для z-panel наиболее согласованный с текущей кодовой базой путь — **не переносить функциональность 3x-ui внутрь репозитория**, а позиционировать z-panel как **клиентский сетевой слой** для Linux и официально описать **совместное использование с 3x-ui** на сервере. Программная «полная интеграция» в одном процессе потребует отдельного ТЗ, аудита **GPL-2/GPL-3** и существенного расширения поверхности атаки и поддержки.
