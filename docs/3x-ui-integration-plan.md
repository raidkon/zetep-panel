# План модуля «z-xray» и паритета с эталоном 3x-ui

**3x-ui в проект не подключается** (ни зависимость `go.mod`, ни встраивание бинарника/репозитория). **[3x-ui](https://github.com/MHSanaei/3x-ui)** используется только как **внешний эталон**: UX (§0.5), перечень возможностей и открытые материалы для сопоставления. Нужный функционал **воспроизводится внутри z-panel** в границах **модуля «z-xray»** (серверная часть + UI вкладки §0.5.1). **Единственная подключаемая внешняя сущность из экосистемы Xray-панелей — это [Xray-core](https://github.com/XTLS/Xray-core)** (`github.com/xtls/xray-core` или запуск внешнего бинарника — по реализации): управление процессом и подача ему **сгенерированного JSON** из нашей модели конфигурации (§2.2).

Документ задаёт модульную архитектуру, сопоставление с эталоном 3x-ui, конфигурацию (§2.3), удалённое управление (**ZPCP**, §0.3) и дорожную карту. Факты про 3x-ui проверены по открытым источникам; состояние z-panel — по дереву `/var/www/home/server/z-panel`.

**Продуктовая цель (верхний уровень):** **система управления серверами** с тем же контрактом **локально / по SSH / удалённый агент** (§0.1–§0.3). Функционал уровня «панель Xray» живёт **только** в модуле **«z-xray»**, без смешения с хостовым слоем **«Сеть → Прочее»** (`xray-redirect`, `xray-tun`, UFW — существующий z-panel).

## 0. Целевая система управления серверами

### 0.1. Единая семантика: локально и по SSH

**Требование:** все функции продукта (конфигурация, **модуль z-xray** / **Xray-core**, хостовая **Сеть** и др.) должны в конечном виде иметь **согласованное поведение** в двух базовых режимах:

- **Локальный режим** — выполнение на том хосте, где установлен бинарник/агент и действует канонический **`/etc/z-panel/config.toml`** (или согласованный эквивалент для платформы).
- **Удалённый режим по SSH** — те же операции адресуются выбранному серверу через транспорт SSH (`internal/transport`, флаги `--ssh=` / `--ssh-connect=`) так, чтобы пользователь не оперировал «двумя разными продуктами», а выбирал **контекст исполнения** (**этот хост** / **хост X по SSH**).

Ограничения текущей кодовой базы (часть команд в удалённом режиме запрещена, см. §2) следует трактовать как **зазор относительно этой цели**: дорожная карта должна его сужать, а не закреплять как постоянную модель.

### 0.2. Панель на другом сервере и «локальный» UX

**Сценарий:** на сервере **B** развёрнут **агент/панель** (свой конфиг и состояние на **B**). На рабочей станции **A** администратор открывает **локальный интерфейс** (CLI и/или будущий UI), в каталоге узлов выбирает **B** и дальше работает так, будто управляет панелью «здесь», при этом:

- **исполнение команд и изменение состояния** выполняются **на B** (на стороне удалённого агента);
- компонент на **A** выполняет роль **клиента управления**: ввод, отображение, маршрутизация запросов, опционально кэш метаданных — **без подмены** привилегированного исполнения на **B**.

Для этого нужен явный **протокол между клиентом (A) и агентом панели (B)** (§0.3). Транспорт может использовать SSH как доставку поверх уже установленного доверия **или** отдельный слушающий сервис на **B**; в обоих вариантах сохраняется правило: **выбрал сервер → эффекты на этом хосте**.

### 0.3. Протокол удалённого управления и сквозное шифрование

Между клиентом управления и агентом на целевом сервере задаётся **отдельно специфицируемый протокол** (рабочее имя на этапе проектирования — **ZPanel Control Protocol, ZPCP**). Для него **обязательны**:

| Свойство | Смысл |
| -------- | ----- |
| **Сквозное шифрование (E2E)** | Содержимое управляющих сообщений (команды, параметры, чувствительные ответы) не передаётся открытым текстом. Если носитель — **SSH**, криптографический контур SSH считается достаточным для транспорта **при явной политике доверия к ключам хоста**. Если носитель — отдельный **TCP/QUIC**, поверх него используется **TLS 1.3** с взаимной аутентификацией **или** протокол класса **Noise** с обменом ключами вне открытого канала. Терминация TLS только на reverse proxy **без** криптографической связи «клиент ↔ агент» не считается выполнением требования E2E для этого слоя. |
| **Аутентификация и авторизация** | Привязка сессии к субъекту (ключи, токены, пароль и MFA по ТЗ), разделение прав по классам операций на агенте. |
| **Целостность и защита от повтора** | Версии сообщений, nonce / монотонные счётчики, привязка к идентификатору сервера. |
| **Согласование возможностей** | Handshake с перечнем поддерживаемых подкоманд и версий схемы, чтобы клиент не отправлял на агент неподдерживаемые запросы. |

**Связь с локальным daemon (§2):** Unix‑сокет остаётся **локальным IPC** для одного хоста; сетевой протокол **не ослабляет** §2.1: секреты в конфиге — по политике хэшей; ключи для mTLS/Noise хранятся и обрабатываются без попадания в логи.

### 0.4. Модульная структура и граница «z-xray»

- **Модуль «z-xray»** — единственное место реализации функционала уровня **3x-ui** (inbounds, клиенты, шаблоны маршрутизации Xray, DNS, балансировщики, обзор метрик трафика и т.д. — по ТЗ и матрице §3). Код и данные модуля **изолированы** от CLI **`xray-redirect` / `xray-tun` / `ufw`** (это остаётся **хостовым сетевым слоем** z-panel без обязательной связки с процессом Xray).
- **Внешняя зависимость:** только **Xray-core** — запуск процесса, перезапуск, выдача **JSON-конфига**, совместимого с ядром. Никакого импорта или линковки с **3x-ui**.
- **Реализация «как у 3x-ui»** — по спецификации и тестам паритета, **без копипасты** GPL-3 кода 3x-ui; при изучении эталона — юридически осознанный **clean-room** / собственная реализация (риск в §4).
- Каталог серверов (§0.2) содержит **узлы с агентом z-panel**, а не «тип 3x-ui»; удалённое управление — **ZPCP** / SSH к **нашему** API агента, не HTTP к чужой панели.

### 0.5. Навигация веб-панели (целевая IA)

В **[3x-ui](https://github.com/MHSanaei/3x-ui)** в боковом меню (`web/html/component/aSidebar.html`) задаются пункты: **дашборд**, **подключения** (inbounds), **настройки** панели, **настройки Xray**, **выход** — фокус на одну серверную установку **Xray**. Мы **не** воспроизводим эту карту один в один: в z-panel отдельно **модуль «z-xray»** (§0.4) и хостовая **«Сеть → Прочее»**.

Целевая панель продукта **z-panel** (веб-слой поверх агента и сценариев §0) должна опираться на **мультисерверный контекст** и **сетевой слой**, а не дублировать карту 3x-ui один в один. Элементы верхнеуровневой навигации (или шапки):

| Элемент UI | Назначение |
| ---------- | ---------- |
| **Выпадающий список серверов** | Выбор узла из каталога (локальный агент, удалённые хосты по **ZPCP**/SSH и т.д.); все экраны ниже относятся к **выбранному** серверу, если не оговорено иное. |
| **Профиль** | Учётная запись и сессия; **подзаголовок/подпись** — **текущий пользователь** (логин, роль или отображаемое имя), чтобы всегда было видно, под кем выполняется работа. Выход из системы логично разместить **внутри** раздела профиля или в выпадающем меню профиля (отдельный пункт «Выход» в корне меню не обязателен, если выход доступен из профиля). |
| **Сеть** | Политики и состояние сети на выбранном узле; см. подразделы ниже. |
| **Настройки** | Параметры приложения панели и агента (язык, внешний вид при наличии, пути, интеграции), в том числе соответствующие канону **`/etc/z-panel/config.toml`** там, где это отображается пользователю. |

**Подразделы раздела «Сеть»:**

| Подраздел | Назначение |
| --------- | ---------- |
| **z-xray** | Весь функционал этой ветки реализуется **модулем «z-xray»** (§0.4): обзор, подключения, настройки Xray по вкладкам §0.5.1; данные и API модуля — в каноне z-panel (§2.2), процесс — **Xray-core**. |
| **Прочее** | Функции уровня ОС и маршрутизации в духе текущего **z-panel**: `xray-redirect`, `xray-tun`, UFW-хелперы, состояние интерфейсов — по мере реализации в UI аналоги CLI-команд (при необходимости дробятся на отдельные подразделы по ТЗ). |

#### 0.5.2. Стек фронтенда

- **Фреймворк:** **Angular** последней стабильной версии на момент реализации веб-клиента (обновления по мере выхода LTS/релизов).
- **Zone.js:** **не используем** — сборка и bootstrap приложения в **zoneless**-режиме: не подключаем `zone.js` в `polyfills` / entry, подключаем провайдеры change detection для zoneless согласно **официальной документации выбранной major-версии** Angular.
- **Signals — везде:** реактивное **состояние и производные значения** в клиенте задаём через **Angular Signals** (`signal`, `computed`, при необходимости `linkedSignal`, **`resource`** для async-данных): выбранный сервер, данные экранов, флаги загрузки и ошибок, условная видимость. Шаблоны читают сигналы через вызовы `()` / control flow; **не** опираемся на хранение «голых» полей класса как единственный источник правды без сигналов. Транспорт с сервером (HTTP и т.д.) по возможности приводим к сигналам через **`resource`**, **`toSignal()`** или явное `update`/`set` по результату запроса; **RxJS** — точечно (эффекты, операторы), без превращения всего слоя в цепочки `Observable` там, где достаточно сигналов.
- **Следствие zoneless + signals:** внешние события, таймеры и колбэки сторонних библиотек должны сходиться в обновление сигналов или использовать API, совместимые с zoneless (без ожидания неявного tick от Zone.js).

## 2. Текущее состояние z-panel

### 2.1. Архитектурное требование: единый файл конфигурации и секреты

**Канонический источник настроек z-panel** — **один файл** **`/etc/z-panel/config.toml`** (`internal/config/config.go`). Все сохраняемые параметры продукта и **снимок состояния `xray-redirect`** (для `down`) хранятся **в этом файле** (секция **`[xray_redirect.<iface>]`** / карта `xray_redirect` в TOML). **Отдельно от канона** допускаются только **временные файлы среды выполнения**: Unix-сокет **daemon** (`socket_path`, по умолчанию под `/run/z-panel/`) и **PID-файл** процесса daemon (`pid_path`, по умолчанию `/run/z-panel/z-panel.pid`) — они **не** дублируют настройки, а отражают запущенный процесс.

**Настройки z-panel задаются только `config.toml`:** переменных вида `Z_PANEL_*` нет. Язык при `language = auto` берётся из обычных системных переменных локали (**`LANGUAGE`**, **`LC_ALL`**, **`LC_MESSAGES`**, **`LANG`**), без отдельного префикса z-panel. Режим **`z-panel --ssh=…`** читает локальный **`config.toml`** для `ssh_no_tty`, `ssh_no_multiplex`, `no_banner` и т.д.; целевой хост задаётся только аргументами CLI.

**Секреты** (пароли, долгоживущие токены и аналоги, если они появятся в конфигурации продукта): **требование** — в файле хранить только **хэши** с современными схемами вроде **bcrypt** или **Argon2** (выбор параметров и точный формат полей — задача реализации; открытый текст в конфиге не допускать).

**Отличие от эталона 3x-ui:** там учёт inbounds и панели живёт в **SQLite**. **z-panel** по-прежнему **без СУБД для канона** (§2.1): домен модуля **«z-xray»** (учётные записи панели, inbounds, клиенты, лимиты — по ТЗ) хранится в **расширенном каноне** в **`config.toml`** (и при необходимости согласованных доп. ключах того же файла), с **проекцией** в JSON для **Xray-core**. Снимок поднятого **`xray-redirect`** хранится **в том же `config.toml`** под **`xray_redirect`** (не отдельный каталог `state`).

### 2.2. Канон z-panel, модуль «z-xray» и JSON для **Xray-core**

В эталоне **3x-ui** итоговая конфигурация **Xray** — **JSON по схеме Xray**, собираемый из **SQLite** и форм. В **z-panel** **3x-ui не используется**; модуль **«z-xray»** (§0.4) держит предметные данные в **нашем каноне** и **сам** строит JSON для **единственной внешней зависимости этого модуля — [Xray-core](https://github.com/XTLS/Xray-core)**.

| Слой | Роль |
| ---- | ---- |
| **Канон z-panel** | Один **`config.toml`**: агент, хостовая сеть, daemon, язык, **`xray_redirect`** (§2.3.4) — §2.3.2. Расширения модуля **«z-xray»** по ТЗ — в **том же файле** (без SQLite). |
| **Модуль «z-xray»** | Читает/пишет только канон; отвечает за жизненный цикл **Xray-core**, валидацию, генерацию **runtime JSON**; публичный HTTP/gRPC API модуля для Angular UI и для **ZPCP** (§0.3). |
| **Проекция → Xray** | JSON на диске или stdin для процесса Xray — **строго односторонняя** «модель z-panel → `config.json` Xray». Сырой JSON **не** является вторым каноном; ручное редактирование на диске вне агента — внешняя деградация, восстанавливаемая повторным `apply` из канона. Импорт чужого `config.json` — **миграция** с разбором в типы z-panel. |
| **Кэш / диагностика** | Временные файлы (например последний JSON Xray в `/tmp`) **только** для диффов и отладки; не канон. |
| **Хостовый слой** | `xray-redirect`, `xray-tun`, `ufw` — **вне** модуля «z-xray»; общий `config.toml` §2.3.2, без обязательной связи с процессом Xray. |

**Angular (§0.5.2):** формы вкладок «z-xray» работают с **API модуля** и доменными DTO z-panel; превью сгенерированного JSON Xray — **read-only** или экспорт.

**Конфликт с правками на диске:** если JSON Xray меняют вручную на хосте, при следующем управлении через агент состояние **перезаписывается** из канона (или явный режим «вне управления» — по ТЗ).

#### 2.3.1. Константы (не из `config.toml`)

| Имя (Go) | Значение по умолчанию | Смысл |
| -------- | --------------------- | ----- |

#### 2.3.2. Файл `/etc/z-panel/config.toml` — структура `Cfg` (`internal/settings/settings.go`)

Все ключи верхнего уровня — в одной корневой таблице TOML; **`xray_redirect`** — вложенная карта (подтаблицы по интерфейсу). Пустая строка для строковых полей после загрузки заменяется значением по умолчанию (`normalize`), кроме языка.

| Ключ TOML | Тип | Значение по умолчанию | Где используется / смысл |
| --------- | --- | --------------------- | ------------------------ |
| `schema_version` | целое | `0` в файле → при чтении legacy трактуется как **1**; актуальная версия в коде — **`CurrentSchemaVersion` = 4** (`internal/settings/schema.go`) | Миграции при обновлении (в т.ч. перенос снимков из старого каталога `state/` при первом `Load`, см. `mergeLegacyStateFiles`). |
| `table` | строка | `51843` | Таблица маршрутизации / fwmark для **xray-redirect**. |
| `systemd_network_dir` | путь | `/etc/systemd/network` | Drop-in **systemd-networkd** для **xray-tun**. |
| `ufw_marker` | строка | `z-panel` | Маркер **UFW**. |
| `xray_tun_managed_mark` | строка | `# z-panel-managed` | Маркер **xray-tun**. |
| `default_lan_cidr` | CIDR | `192.168.0.0/16` | LAN по умолчанию. |
| `default_lan_dev` | имя iface | `lan0` | Интерфейс LAN по умолчанию. |
| `default_xray_addr` | CIDR | `10.252.0.1/30` | Адрес TUN по умолчанию. |
| `default_xray_peer` | CIDR | `10.252.0.2/30` | Peer TUN по умолчанию. |
| `socket_path` | путь | `DefaultSocketPath` | Unix-сокет **daemon** (отдельно от `config.toml`). |
| `pid_path` | путь | `DefaultPidPath` | PID-файл **daemon** (отдельно от `config.toml`). |
| `xray_redirect` | таблицы | — | Снимки **`xray-redirect up`** по ключу интерфейса (§2.3.4). |
| `daemon` | `0` или `1` | `0` | Пересылка подкоманд на сокет (`internal/client`). |
| `no_banner` | bool | `false` | Не печатать первую строку stderr с версией при каждом запуске. |
| `ssh_no_multiplex` | bool | `false` | Для `z-panel --ssh=…`: не поднимать SSH ControlMaster (без общего `-S` сокета). |
| `ssh_no_tty` | bool | `false` | Для `z-panel --ssh=…`: не передавать `ssh -t` (удобно при NOPASSWD sudo). |
| `language` | строка | `auto` | Явный язык UI (`en`, `ru`, …) или **`auto`** — тогда используются системные **`LANGUAGE`**, **`LC_ALL`**, **`LC_MESSAGES`**, **`LANG`** (без переменных `Z_PANEL_*`). |

Интерактивный **`z-panel config init`** запрашивает: `language`, `table`, `systemd_network_dir`, `default_lan_cidr`, `default_lan_dev`, `default_xray_addr`, `default_xray_peer`, `ufw_marker`, `xray_tun_managed_mark`. Поля **`daemon`**, **`no_banner`**, **`ssh_no_multiplex`**, **`ssh_no_tty`**, **`socket_path`**, **`pid_path`**, **`xray_redirect`**, **`schema_version`** в диалоге не спрашиваются — задаются вручную или программно.

#### 2.3.3. Локаль ОС и `language = auto`

Для **`language = auto`** язык сообщений определяется только стандартными переменными локали процесса: **`LANGUAGE`** (список через `:`), затем **`LC_ALL`**, **`LC_MESSAGES`**, **`LANG`**. Отдельных переменных `Z_PANEL_*` для языка нет.

Флаги **`--ssh=` / `--ssh-connect=`** на корне бинарника задают удалённый режим (`internal/transport`) — это **аргументы** командной строки, не дубликат `config.toml`.

#### 2.3.4. Снимок `xray-redirect` в `config.toml` (`xray_redirect`, `internal/state`, `internal/settings/xray_redirect.go`)

После **`xray-redirect up`** состояние сохраняется **в том же** `config.toml` в секции **`[xray_redirect.<ключ_интерфейса>]`** (ключ санитизируется: `/` и `.` → `_`). Поля совпадают с прежней JSON-структурой:

| Поле | Тип | Смысл |
| ---- | --- | ----- |
| `mode` | строка | Режим (опционально). |
| `interface` | строка | Имя интерфейса. |
| `table` | строка | Таблица/политика. |
| `fwmark` | строка | Fwmark. |
| `wg_ipv6` | bool | IPv6. |
| `no_bypass_mark` | bool | Без bypass mark. |
| `bypass_cgroup` | строка | Cgroup для bypass. |
| `bypass_unit` | строка | Systemd-юнит bypass. |
| `wan_rule_pref` | int | Приоритет правила WAN. |
| `wan_from_cidr` | строка | CIDR для политики «с WAN». |

**Legacy:** при первом `Load` снимки из каталога **`/etc/z-panel/state/*.json`** импортируются в память и подмешиваются в `xray_redirect`, если карта была пуста; при следующем **`Write`** остаются только в TOML. Файлы **`socket_path`** и **`pid_path`** не являются частью канона данных.

#### 2.3.5. Планируемые расширения `config.toml` (пока не в коде)

Согласовано с §0, §2.1–§2.2; конкретные имена ключей — на этапе реализации и поднятия `schema_version`.

| **Домен «z-xray»** | Расширения **в том же** `config.toml` для inbounds, клиентов, лимитов (§2.2). |
| **Транспорт ZPCP** | Порты, профили доверия, таймауты handshake — расширение канона без второго «равного» файла. |
| **Веб-клиент / UI** | URL API агента по умолчанию, тема — при необходимости **в том же** `config.toml`. |
| **Генерация конфига Xray** | Внутри модуля «z-xray»; формат хранения — термины z-panel, проекция в JSON Xray-core (§2.2). |

| Аспект                 | Содержание                                                                                                                                                                                                                                                                                                       |
| ---------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Назначение**         | **CLI для Linux** для настройки **политик маршрутизации через TUN** в стиле wg-quick: дефолтный маршрут через выбранный интерфейс, опциональный обход через **cgroup**, правила фаервола против утечек, помощники для **UFW** и адресации **systemd-networkd** для TUN. Канон — **файлы** (§2.1–§2.2), **не SQLite** и не зависимость от репозитория **3x-ui**.

## 4. Поэтапный план внедрения

### Фаза 0 — Продуктовое решение и границы

- Зафиксировать **модульную** архитектуру (§0.4): **«z-xray»** vs хостовая **«Сеть → Прочее»**; **не** подключать **3x-ui**; внешняя зависимость домена Xray — только **Xray-core**.
- Юридически: стратегия **clean-room** относительно GPL-3 кода **3x-ui** (эталон по документации и поведению, не копипаста); лицензия z-panel **GPL-2+** без смешивания репозиториев.
- Продуктовая цель §0: **ZPCP**, каталог узлов, мультисерверный UX.
- Модель угроз: daemon и будущий API «z-xray» (§0.3).

### Фаза 1 — Документация и контракты

- Спецификация **канона** для домена «z-xray» (расширение §2.3.5, §2.2): форматы файлов, `schema_version`.
- Документация для пользователей: **без** sidecar 3x-ui; связка **модуль z-xray + Xray-core** и существующий **xray-redirect** на клиенте при необходимости.
- Описать **ZPCP** / SSH и env (§2.3.3); опубликовать §2.3.

### Фаза 2 — Хостовая эксплуатация (текущий z-panel)

- Unit-файлы / порядок запуска **Xray-core** (управляемый модулем) и **xray-redirect** на клиентских сценариях.
- Чеклисты утечек (маршруты, cgroup, UFW).

### Фаза 3 — Модуль «z-xray» и удалённое управление

- Реализация ядра модуля: хранение канона, генерация JSON, управление **Xray-core**, API для Angular.
- **ZPCP** и сценарий «UI на A → агент на B».
- Аутентификация API (выход из модели «только root + сокет» для сетевого доступа).

### Фаза 4 — Расширение модулей (по ТЗ)

- Дополнительные модули (Telegram, LDAP, …) **вне** ядра «z-xray» или как подключаемые части с явными границами.
- Сохранять **без СУБД для канона** (§2.1), если иное не оговорено отдельным архитектурным решением.

### Зависимости между фазами

```text
Фаза 0 (архитектура, лицензии)
    ↓
Фаза 1 (спецификации, канон «z-xray»)
    ↓
Фаза 2 (хостовой z-panel + Xray в эксплуатации)
    ↓
Фаза 3 (модуль «z-xray», ZPCP, UI)
    ↓
Фаза 4 (опционально: доп. модули)
```

### Риски


| Риск | Митигация |
| ---- | --------- |
| Путаница модулей («z-xray» vs хостовая сеть) | Документация §0.4, раздельные пакеты/API в коде. |

| Ошибки модели угроз для ZPCP | Угрозный анализ до включения по умолчанию (§0.3). |

## 5. Целевая архитектура (один продукт z-panel)

Монолит **3x-ui** на сервере **не** используется. На узле — **агент z-panel**: **модуль «z-xray»** (управление **Xray-core** + канон в файлах) и **хостовые** команды (`xray-redirect`, `xray-tun`, `ufw`). На клиенте Linux — тот же z-panel для политики маршрутизации при необходимости.

```text
┌─────────────────────────────────────────────────────────────────┐
│                   Сервер (VPS) — агент z-panel                   │
│  ┌──────────────────────┐         ┌─────────────────────────┐   │
│  │ Модуль «z-xray»      │ ──────► │ Xray-core               │   │
│  │ (канон → JSON)       │  конфиг │ inbounds / routing / …  │   │
│  └──────────────────────┘         └─────────────────────────┘   │
│  ┌──────────────────────┐                                        │
│  │ Хостовая сеть        │  маршруты, UFW, systemd-networkd     │
│  │ xray-redirect/tun…   │  (без обязательной связи с модулем)  │
│  └──────────────────────┘                                        │
│  config: /etc/z-panel/… ; ZPCP для удалённого UI                  │
└─────────────────────────────────────────────────────────────────┘
                              │
                              │ VPN/TUN трафик (по сценарию)
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│              Клиент (Linux) — z-panel при необходимости         │
│         xray-tun / xray-redirect / ufw ; канон в config.toml │
└─────────────────────────────────────────────────────────────────┘
```

**Дополнение — мультисерверное управление (§0):** рабочая станция администратора **A** держит локальный UI; выбранный узел **B** выполняет команды на своём агенте; канал **A ↔ B** защищён по правилам **ZPCP** (или эквивалент SSH-транспорта при явной политике ключей).

```text
  ┌──────────────────────┐         ZPCP / SSH           ┌──────────────────────┐
  │  Администратор (A)    │  ─────── (E2E по §0.3) ────► │  Сервер B: агент     │
  │  локальный UI         │                             │  z-panel / панель    │
  └──────────────────────┘                             └──────────────────────┘
           │                                                      │
           └──────── исполнение запросов только на стороне B ─────┘
```

**Границы данных:**


| Компонент | Данные |
| --------- | ------ |
| **Модуль «z-xray»** | Канон в файлах под **`/etc/z-panel/`** (расширение §2.3.5): inbounds, клиенты, лимиты, учёт панели; **проекция** в JSON для **Xray-core**. |
| **Xray-core** | JSON-конфигурация от агента, runtime соединений. |
| **Хостовый z-panel** | `config.toml` §2.3.2 (`xray_redirect` §2.3.4); runtime: сокет и PID §2.3.2. |


**Границы доверия:** веб/UI и **ZPCP** — **сетевые** поверхности с TLS/MFA по ТЗ; daemon — **локальный root IPC** до появления сетевого API; секреты — §2.1.

## 6. Итоговая рекомендация

Проект **не подключает 3x-ui**. Функционал уровня панели Xray переносится **модулем «z-xray»** с **модульной** архитектурой и **единственной** внешней зависимостью этого домена — **[Xray-core](https://github.com/XTLS/Xray-core)**. Эталон **[3x-ui](https://github.com/MHSanaei/3x-ui)** используется для **паритета UX и возможностей** (§0.5, §3) при **собственной** реализации и учёте лицензий (§4 фаза 0). Хостовые возможности текущего z-panel сохраняются отдельно от модуля «z-xray» (§0.4). Удалённое управление — **ZPCP** и каталог узлов (§0.1–§0.3).
